# Release Notes

This document explains how we manage release notes (changelogs) for the Panfactum project.

## Overview

Every release gets a structured YAML entry (`log.yaml`) that describes its changes. These entries
are validated against a JSON Schema at commit time and rendered into the changelog pages on
[panfactum.com](https://panfactum.com/docs/changelog/edge/0).

The system is designed so that:

- Changes are machine-readable and queryable (structured YAML, not free-form prose).
- Impact tracking is first-class — every change can declare which IaC modules, CLI commands,
  devshell tools, or configuration files it affects.
- Validation catches errors before they reach `main`.

## Directory Layout

All changelog content lives under:

```
packages/website/src/content/changelog/
```

The directory is organized by release channel:

```
changelog/
├── log.schema.json              # Generated JSON Schema (do not edit by hand)
├── main/
│   ├── log.yaml                 # Unreleased changes for the next edge release
│   └── upgrade.mdx              # (optional) Upgrade instructions for the next release
├── edge/
│   ├── YY-MM-DD/
│   │   ├── log.yaml             # One entry per edge release
│   │   └── upgrade.mdx          # (optional) Upgrade instructions
│   └── ...
└── stable/
    ├── YY-MM/
    │   ├── summary.mdx          # Channel-level overview
    │   ├── upgrade.mdx          # Channel-level migration guide
    │   ├── 0/
    │   │   └── log.yaml         # Patch release 0
    │   ├── 1/
    │   │   └── log.yaml         # Patch release 1
    │   └── ...
    └── ...
```

### Release Channels

| Channel  | Naming     | Purpose                                          |
|----------|------------|--------------------------------------------------|
| `main`   | (single)   | Unreleased work — add changes here as you go.    |
| `edge`   | `YY-MM-DD` | Rapid releases cut from `main`.                  |
| `stable` | `YY-MM`    | Long-term support branches with numbered patches. |

## Where to Make Changes

**Almost all changelog work happens in `main/log.yaml`.** When you make a code change that should
appear in the release notes, add it to `packages/website/src/content/changelog/main/log.yaml`.
That file accumulates unreleased changes until the next edge release is cut.

### `edge/` — Do Not Edit Directly

The `edge/` directories are created by automated release scripts (`make-new-edge-release`). When
a release is cut, the scripts move the contents of `main/` into a new `edge/YY-MM-DD/` directory
and reset `main/log.yaml` for the next cycle.

**Do not create or edit files in `edge/` unless you are fixing up a released entry** (e.g.,
correcting a typo or adding a missing reference after the fact).

### `stable/` — Rare Manual Edits

Stable channels are created by `make-new-stable-release-channel`. Patch releases within a stable
channel may require manual edits, but this is uncommon. If you need to add a change to a stable
patch, edit the specific `stable/YY-MM/N/log.yaml` file directly.

## The `log.yaml` Format

Every `log.yaml` file must conform to the JSON Schema at
`packages/website/src/content/changelog/log.schema.json`. Refer to that file for the full
field definitions, types, required/optional status, and allowed enum values.

The valid `component` values in each `impacts` entry are auto-generated from the codebase (see
[Schema Generation](#schema-generation) below). If you reference a component that doesn't exist
in the schema, validation will fail.


## Upgrade Instructions (MDX Files)

When a release has non-trivial migration steps, create an `upgrade.mdx` file alongside the
`log.yaml` and set `upgrade_instructions: upgrade.mdx` in the YAML.

These files are standard MDX and can use the site's components:

```mdx
{/* Upgrade instructions for edge release YY-MM-DD. */}

import MarkdownAlert from "@/components/markdown/MarkdownAlert.astro"

<MarkdownAlert severity="warning">
  Important caveat about this release.
</MarkdownAlert>

## Step 1: Update Provider Versions

All terraform provider versions have been upgraded. Update your first-party modules...

## Step 2: Component Upgrades

Review each component's release notes:

- [Kubernetes 1.29 → 1.30](https://github.com/kubernetes/kubernetes/blob/master/CHANGELOG/CHANGELOG-1.30.md)
```

## Schema Generation

The `log.schema.json` file is **not** edited by hand. It is generated by:

```
ds-generate-changelog-schemas
```

This script introspects the codebase to build the valid `component` enums dynamically:

| Impact Type     | Source of Valid Values |
|-----------------|----------------------|
| `iac-module`    | Directory names under `packages/infrastructure/`, plus old/removed names from `packages/infrastructure/metadata.yaml` |
| `cli`           | `static override paths` declarations in `packages/cli/src/commands/**/command.ts` |
| `devshell`      | `.sh` scripts in `packages/local/scripts/` and `packages/devshell/scripts/`, plus a static list of well-known nix binaries |
| `configuration` | A static list of standard Panfactum config files (e.g., `panfactum.yaml`, `region.yaml`) |

### Runtime Validation

The Astro content collection in `packages/website/src/content.config.ts` also validates the
YAML data at build time using a Zod schema that mirrors the JSON Schema. This provides type-safe
access in the website code.

## Key File Paths

| Purpose                  | Path |
|--------------------------|------|
| Changelog entries        | `packages/website/src/content/changelog/[channel]/[release]/log.yaml` |
| Upgrade instructions     | `packages/website/src/content/changelog/[channel]/[release]/upgrade.mdx` |
| Log schema (generated)   | `packages/website/src/content/changelog/log.schema.json` |
| Schema generator         | `packages/local/scripts/ds-generate-changelog-schemas.ts` |
| Changelog validator      | `packages/local/scripts/ds-validate-changelog.ts` |
| Astro content config     | `packages/website/src/content.config.ts` |
| Changelog page routes    | `packages/website/src/pages/docs/changelog/[...page].astro` |
| Pre-commit config        | `.pre-commit-config.yaml` |
